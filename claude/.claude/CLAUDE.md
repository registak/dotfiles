# Global Instructions

## 基本方針

- 実装・提案はファクトに基づく。推測で進めない
  - 根拠の優先順位: 公式ドキュメント > ソースコード > 動作確認
  - レファレンスが不十分なら、実装前に調査ステップを設ける
  - 調査しても確信が持てなければ、その旨を伝えて確認を求める
- 複数の情報源が矛盾する場合は鮮度を評価し、最新と判断した根拠を示す
- 外部ライブラリ・APIはバージョンを意識し、公式ドキュメントで裏を取る

## コミュニケーション

- 非自明なタスクでは実装前にプランを提示する。トレードオフ・不明点・設計判断を明示
- 完了報告は結論ファースト — 何がどう変わったかを冒頭に
- Markdownで太字を多用しない。見出し・リストで構造化する
- 冗長な前置き・繰り返しは不要。簡潔に、ただし浅くならない

## 作業スタイル

- 独立した作業はAgentを並列起動して処理する
- 実装を伴うAgentはworktreeで分離する（`isolation: "worktree"`）
- コミットメッセージはConventional Commits形式
- コミットはCIがパスする最小の動作単位で切る
- Issue・PR作成時はリポジトリのテンプレートがあれば使う
- 変更後はビルド・テスト・lintを実行し、壊れていないことを確認してから報告する
