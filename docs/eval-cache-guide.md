# eval キャッシュ化ガイド

zoxide / starship の `eval` を避けてキャッシュファイルから `source` するパターンの解説。

---

## 仕組み

```bash
# eval: 毎回サブプロセスを起動して初期化コードを生成・評価
eval "$(zoxide init zsh)"    # fork + exec + 出力キャプチャ + eval

# キャッシュ: 初回だけ生成し、以降はファイルを読むだけ
_zoxide_cache="$HOME/.cache/zsh/zoxide-init.zsh"
if [[ ! -f "$_zoxide_cache" || "$(which zoxide)" -nt "$_zoxide_cache" ]]; then
  mkdir -p "${_zoxide_cache:h}"
  zoxide init zsh > "$_zoxide_cache"
fi
source "$_zoxide_cache"
```

`-nt`（newer than）でバイナリのタイムスタンプとキャッシュを比較し、`brew upgrade` 等でバイナリが更新されたら自動再生成する。

---

## メリット

### シェル起動速度の向上

| 方式 | 所要時間（目安） |
|------|-----------------|
| `eval "$(zoxide init zsh)"` | ~5-15ms |
| `source` キャッシュ | ~1-2ms |

2ツール合計で 10-30ms 程度の短縮。単体では微小だが、ターミナル分割やスクリプトからの zsh 起動のたびに効く。

### 設計方針の一貫性

| ツール | eval 回避の方法 |
|--------|----------------|
| Homebrew | 環境変数を直接展開 |
| mise | shims PATH |
| zoxide / starship | キャッシュファイル |

zoxide / starship はシェル関数を生成するため直接展開ができず、キャッシュが現実的な妥協点。

---

## デメリット

### コードの複雑化

1行が6行になる。読む人が「なぜキャッシュしているのか」を理解する必要がある。

### キャッシュ不整合のリスク

`-nt` チェックはバイナリのタイムスタンプだけを見る。

- 検知できる: `brew upgrade` でバイナリが更新された場合
- 検知できない: キャッシュファイルが手動で壊された場合（存在するので再生成されない）

---

## 注意点

### キャッシュの手動リセット

挙動がおかしいときはキャッシュを消す。次のシェル起動で自動再生成される。

```bash
rm -rf ~/.cache/zsh/
```

### 設定ファイルの変更は影響しない

- `starship.toml` → starship はプロンプト描画時に設定を読むので、init 出力はテーマに依存しない
- zoxide → init は `z` / `zi` の関数定義のみで、設定変更で init 出力は変わらない

この2ツールについてはキャッシュの実害がほぼないケース。

---

## まとめ

| | eval | キャッシュ |
|---|---|---|
| 速度 | ~15ms/ツール | ~2ms/ツール |
| コードの簡潔さ | 1行 | 6行 |
| 壊れにくさ | 常に最新 | ごく稀にキャッシュ破損 |
| 回復方法 | — | `rm ~/.cache/zsh/` |

「方針の統一」に価値を置くなら入れる、「1行の簡潔さ」を優先するなら eval のままでも実害はない。
